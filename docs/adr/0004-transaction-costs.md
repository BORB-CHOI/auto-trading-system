# ADR-0004: 거래비용·슬리피지 모델

- **상태:** 수락 (1단계 정액률 모델만 — 슬리피지 곡선은 미해결)
- **날짜:** 2026-07-22
- **관련:** PROJECT_GUIDELINES §6.4(거래비용 다단계), §6.1(통계 검증), `screening.py`, CLAUDE.md

## 맥락 (Context)

백테스트가 매매를 재생할 때 **한 번 사고팔며 새는 돈**(수수료·세금·슬리피지)을 빼지 않으면
수익률이 부풀려진다. §6.4 통찰: *"3% 목표 익절인데 수수료·슬리피지로 실제 수익은 0 또는 마이너스."*
CLAUDE.md 도 "거래비용·슬리피지 처음부터 포함"을 못박았다.

두 가지가 이 모델에 걸려 있다.

1. **엔진의 손익 계산** — 매수/매도 때 비용을 물려야 순손익이 나온다.
2. **`screening.py` 의 거래대금 하한** — "거래대금 작은 종목일수록 슬리피지가 크다"가 하한의 근거다.
   즉 하한은 슬리피지 모델이 정한다.

이 둘의 요구 정밀도가 다르다. ①은 §6.4 의 정액률 시나리오로 충분하지만, ②는 **주문 크기 대비
거래대금**에 반응하는 곡선이 있어야 한다. 정액률은 종목 크기와 무관해서 하한을 못 정한다.

## 검토한 선택지 (Options)

**A. §6.4 정액률 다단계만 (수준 0~3).**
- 장점: §6.4 에 이미 사양이 있음. 정밀 회계가 아니라 **시나리오 스트레스 손잡이**라 단순·투명.
  과거 데이터 없이 입력→출력으로 단위 검증 가능. 엔진(①)을 가장 빨리 언블록.
- 단점: 종목 크기와 무관 → `screening.py` 거래대금 하한(②)을 이걸로는 못 정한다.
- 가드레일: 위반 없음. 전부 placeholder, 하드코딩 아님.

**B. 처음부터 주문크기 의존 슬리피지 곡선 + 매수/매도 비대칭 + 거래세 시점 스케줄.**
- 장점: ②까지 한 번에. 더 현실적.
- 단점: 곡선 형태·계수가 전부 추정이라 지금 근거 없이 숫자를 박게 된다(하드코딩 위험).
  거래세는 2017~현재 여러 번 바뀌어 시점별 스케줄 검증이 필요 — 범위가 커진다.
  엔진이 없어 곡선을 **검증할 수단이 아직 없다**(백테스트 P&L 대조 불가).
- 가드레일: 근거 없는 계수 확정은 §0.1(placeholder 원칙) 취지에 어긋날 소지.

## 결정 (Decision)

**A 를 1단계로 채택.** §6.4 정액률 다단계 모델을 `src/layer4_execution/costs.py` 에 구현한다.

- `CostModel(round_trip_rate)` — 왕복(매수+매도) 총 비용률, 체결금액 대비. **전부 placeholder.**
- §6.4 수준(placeholder 시나리오): 0=0.0% / 1=0.3% / 2=0.5% / 3=0.8%.
- 백테스트 **기본값 = 수준 2(0.5%)**. 수준 1 은 sanity check, 수준 3 은 stress.
- `net_return(g) = g − round_trip_rate` — §6.4 통찰("비용이 익절을 잡아먹는다")을 그대로 인코딩.
- 정량값은 전부 placeholder. 근거(실측 슬리피지 등)가 생기면 바꾼다.

**B 의 요소는 명시적으로 뒤로 미룬다.** 엔진이 서서 P&L 대조가 가능해진 뒤 다음 벽돌로 다룬다.

## 결과 (Consequences)

- 백테스트 엔진(다음 작업)은 `DEFAULT_COST` 를 왕복 1회당 물리는 것으로 시작할 수 있다.
- §6.4 채택 기준(수준 2 양수=minimum, 수준 3 양수=robust)은 **expectancy 가 나온 뒤**(엔진+§6.1
  통계) 판정 함수로 붙인다. 지금은 `COST_LEVELS` 목록만 둔다.
- `screening.py` 거래대금 하한은 **아직 정하지 못한다** — 미해결 1이 풀려야 한다.
- 재검토 조건: 실측 슬리피지가 정액률 가정과 크게 다르면 이 ADR 을 다시 연다.

## 미해결 (Open questions)

1. **주문 크기 의존 슬리피지 곡선** — 거래대금 대비 주문 비중 → 슬리피지 bp. `screening.py`
   거래대금 하한을 정하는 핵심. 곡선 형태(제곱근 충격 모델 등)와 계수를 뭘로, 어떻게 검증할지 미정.
2. **매수/매도 비대칭** — 한국은 거래세가 매도에만 붙는다. 지금은 왕복 합산으로 뭉갠다.
   엔진이 매수/매도를 나눠 물리려면 per-side 비용으로 분리해야 한다.
3. **거래세 시점 스케줄** — 거래세율이 2017~현재 여러 차례 변경. 정확한 시점·세율은 공식 출처
   (국세청·KRX) 대조가 필요하다. 지금 정액률은 이를 흡수한 근사다. (오너 확인 필요)
4. **호가/체결 가정** — §6.4 는 "시장가 ❌, 지정가 + 호가창 슬리피지". 지정가 미체결(못 사고 지나감)을
   백테스트가 어떻게 처리할지 미정.
