# ADR-0005: 케이스 검사기 웹 도구 (시각 도구 아키텍처)

- **상태:** 수락 (1단계 = 캔들차트 슬라이스 완료)
- **날짜:** 2026-07-22
- **관련:** PROJECT_GUIDELINES §4.2(screening), §6.1(통계 검증), CLAUDE.md(매매 실행=결정론적 코드), BORB-31

## 맥락 (Context)

오너가 방향을 정했다. **"통상적인 전구간 백테스트"가 아니라**, 특정 종목·시점을 골라
**차트로 보고 → (조건검색식) 종목 선별 → 내가 정한 매매 전략을 얹어** "이 케이스가 어땠나"를
확인하는 **웹 도구**를 원한다. 순서: ① 차트 → ② 조건검색 종목 선별 → ③ 전략 오버레이.

이건 지침서가 세운 가드레일 백테스트(survivorship bias·point-in-time·train/test)와 **목적이 다르다**.
그건 *검증*이고, 이 도구는 *탐색·가설 생성*이다. 둘은 대체가 아니라 보완이다.

## 검토한 선택지 (Options)

**A. 파이썬 로컬 차트 (mplfinance/plotly).**
- 장점: 기존 파이썬 코드에 바로. 빠름.
- 단점: 인터랙션(줌·드래그·오버레이 조작)이 약하다. 오너가 원한 "웹"이 아니다.

**B. 웹 SPA (Vite + React + TypeScript) + FastAPI 백엔드.**  ← 채택
- 장점: **AI(작업자)가 가장 안정적으로 짜는 스택**. FastAPI 가 기존 파이썬 데이터 레이어
  (marcap 로더·exclusions)를 **단일 정본으로 재사용**. 인터랙션·확장(조건검색·오버레이) 용이.
- 단점: 순수 파이썬 리포에 JS/웹 스택이 추가된다(구조 변경). 새 의존성.
- 가드레일: 위반 없음. **웹은 그리기만**, 전략·매매 판단은 파이썬(결정론적)에 남긴다.

**C. Next.js.**
- 단점: App Router 등 프레임워크 고유 함정이 많고 이 용도엔 과하다(데이터는 어차피 파이썬 API).

## 결정 (Decision)

**B 채택.** `web/`(Vite+React+TS) + `api/`(FastAPI). 파이썬은 데이터·전략의 정본, 웹은 표현.

- **1단계(완료):** `GET /api/candles`(일봉 JSON, 거래대금 포함) + `GET /api/symbols`(종목 검색).
  `web/vite.config.ts` 가 `/api/*` 를 FastAPI(:8000)로 proxy.
- **차트 라이브러리 변천:** lightweight-charts → KLineChart core → **KLineChart Pro**(완성형 위젯).
  Pro 로 지표 메뉴 전체·이평선(기간 편집)·그리기 툴바(피보나치 등)·심볼 검색이 통째로 들어온다.
  거래대금(turnover)은 커스텀 지표로 등록해 메뉴에 노출. 상승=빨강/하락=파랑, 라이트 테마.
  **주의:** Pro 0.1.1 은 klinecharts v9 전용이라 klinecharts 를 v9 로 고정한다(Pro 유지 비용).
- **가드레일 못박기:** 이 서버·프런트는 BUY/SELL·포지션·주문을 하지 않는다. 전략 평가가 붙을 때도
  로직은 파이썬 layer3 에 두고 프런트는 결과만 그린다.
- **방법론 경계:** 케이스를 눈으로 본 결과는 **가설**이다. 검증이 아니다. 사후확신(이미 결과를 아는
  상태로 과거 차트를 보면 "됐겠네" 착각)을 막기 위해, 재생 기능을 붙일 땐 **미래 봉을 가리고 하루씩
  넘기는** 방식으로 설계한다. 돈을 걸기 전 검증은 여전히 가드레일 백테스트(BORB-31) 몫이다.

## 결과 (Consequences)

- 리포가 파이썬 단일 → **파이썬 + JS 모노레포**로 바뀐다. `docs/ARCHITECTURE.md` 갱신 필요(후속).
- 새 의존성: 파이썬 `webapp` optional group(fastapi·uvicorn·httpx), `web/` npm(react·vite·
  lightweight-charts). 프런트 산출물·node_modules 는 `.gitignore`.
- 실행: 백엔드 `uvicorn api.main:app --port 8000`, 프런트 `cd web && npm run dev`.

## 미해결 (Open questions)

1. **② 조건검색 종목 선별** — 증권사 조건검색을 참고한 선별 UI/로직. layer3 `screening.py`(placeholder)와
   어떻게 잇는가. 오너가 어떤 조건들을 넣을지.
2. **③ 전략 오버레이** — 오너가 정한 전략을 차트 위에 진입/청산 마커로 표시하는 방식. 전략 표현 포맷 미정.
3. **미래 봉 가림 재생 UX** — 사후확신 방지용 재생 모드 설계.
4. **데이터 로딩 성능** — 지금은 연도 parquet 를 `lru_cache` 로 캐시. 종목 인덱스/사전 집계가 필요할지.
5. **배포·실행 범위** — 로컬 전용인가, 나중에 어딘가 올리나. 인증 필요 여부.
